# MHM Doc Writer

Manages Docusaurus documentation updates in `website/docs/` directory, ensuring code changes are reflected in user-facing documentation.

## Important Context

- The `website/` directory is a **separate Git repository** (`MaxHandMade/mhm-rentiva-docs`) nested inside `mhm-rentiva`
- Docusaurus version: **3.9.2** with `future.v4: true`
- Site URL: `https://maxhandmade.github.io/mhm-rentiva-docs/`
- Default locale: **Turkish (`tr`)**
- Sidebar: Auto-generated from filesystem (`autogenerated`, `dirName: '.'`)
- Blog section is used for **Sürüm Notları** (Release Notes)

## Documentation Structure

Documents are categorized under `website/docs/`:
- `01-getting-started/`: Installation and getting started
- `02-core-configuration/`: Core settings
- `03-features-usage/`: Feature usage guides (shortcodes, bookings, vehicles, reports, etc.)
- `04-developer/`: Developer docs, hooks, API
- `05-faq/`: Frequently asked questions

Each category folder has a `_category_.json` file that controls its sidebar label and position.

## Update Workflow

### 1. File Detection
When code changes occur (e.g., new Shortcode added or setting changed), locate or create the corresponding documentation file in `website/docs/`.
- If file doesn't exist, create new `.md` file in appropriate category folder

### 2. Docusaurus Format
Every Markdown file MUST start with **Frontmatter**:

```markdown
---
id: file-id
title: Title (Visible to Users)
sidebar_label: Sidebar Menu Name
---
```

### 3. Content Writing
- **Language:** Primary documentation language is **Turkish**. All titles, descriptions, and menu labels must be Turkish.
- **Style:** Clear, understandable language with correct technical terminology
- **Code Examples:** Always include code blocks (` ```php ... ``` `) with examples

### 4. Update Process
1. **Analysis:** Understand what the changed code does and how it affects users
2. **Target Identification:** Determine which file in `website/docs/` needs updating (or new file path)
3. **Writing:** Update Markdown content
   - Add new parameters to tables if added
   - Update steps if workflow changed
4. **Verification:** Ensure frontmatter structure is intact

## Special Cases

- **New Feature:** Create new file under `03-features-usage/`
- **Dev Change (Hook, Class):** Update under `04-developer/`
- **FAQ Entry:** Add to `05-faq/`
- **Critical Warnings:** Use Docusaurus Admonitions (`:::danger`, `:::warning`, `:::tip`)

```markdown
:::warning
Bu ayarı değiştirmek veritabanı yapısını etkiler. Yedek almadan işlem yapmayın.
:::
```

## Changelog'dan Blog Yazısı Oluşturma

Yeni bir sürüm yayınlandığında `changelog-tr.json` dosyasındaki verileri kullanarak Docusaurus blog yazısı oluşturulmalıdır.

### 1. Veri Kaynağı
`changelog-tr.json` dosyasındaki en üstteki (en yeni) sürüm nesnesini baz al.
- **Structured format:** `{ "type": "fix", "description": "..." }`
- **Legacy format:** `"🐛 HATA DÜZELTMESİ: ..."`

### 2. Dosya Hazırlığı
- **Dizin:** `website/blog/`
- **Dosya Adı:** `YYYY-MM-DD-vX.Y.Z-release.md` (Örn: `2026-02-11-v4.9.9-release.md`)

### 3. Blog Formatı (Şablon)

```markdown
---
slug: vX.Y.Z-release-summary
title: "MHM Rentiva vX.Y.Z: [Sürüm Başlığı]"
authors: [maxhandmade]
tags: [release, update, mhm-rentiva]
date: YYYY-MM-DDT09:00
---

MHM Rentiva **vX.Y.Z** sürümü yayınlandı! Bu güncelleme ile [genel özet cümlesi].

## 🚀 Değişiklik Kaydı

| Tür | Açıklama |
|-----|----------|
| 🛠️ Düzeltme | Örnek hata düzeltmesi açıklaması |
| ✨ Özellik | Yeni özellik açıklaması |

### 4. Teknik Hafıza Entegrasyonu (PROJECT_MEMORIES.md)
Blog yazısını zenginleştirmek için `PROJECT_MEMORIES.md` dosyasını tara:
- **Key Decisions:** Mimari değişikliklerin veya önemli kararların nedenlerini blog yazısının "Detaylar" bölümüne ekle.
- **Files Modified:** Yapılan değişikliklerin teknik kapsamını anlamak için bu listeyi kontrol et.
- **Milestone Linking:** Blog yazısındaki sürümün teknik hafızadaki karşılık gelen `### [LOCKED] ... (vX.Y.Z)` başlığıyla uyumlu olduğundan emin ol.

### 5. Emoji Haritalama
JSON'daki `type` alanlarını Markdown tablosunda emojili başlıklarla eşleştir:
- `fix` -> 🛠️ Düzeltme
- `feature` -> ✨ Özellik
- `improvement` -> ⚡ İyileştirme
- `security` -> 🛡️ Güvenlik
- `stability` -> ⚙️ Kararlılık

## Commit & Deploy Workflow

After updating documentation files, follow these steps:

### 1. Stage and Commit
```bash
cd website/
git add .
git commit -m "docs: <descriptive message>"
```

### 2. Push to GitHub
```bash
git push origin main
```

### 3. Automatic Deployment
- Push to `main` triggers GitHub Actions workflow (`deploy.yml`)
- `peaceiris/actions-gh-pages@v3` builds and deploys to `gh-pages` branch
- `pages-build-deployment` then publishes to GitHub Pages
- Site updates automatically at `https://maxhandmade.github.io/mhm-rentiva-docs/`
- Typical deploy time: ~1-2 minutes

### 4. Verification
After push, check:
- GitHub Actions: `https://github.com/MaxHandMade/mhm-rentiva-docs/actions`
- Live site: `https://maxhandmade.github.io/mhm-rentiva-docs/`

## Media and Screenshots

- Images uploaded to `website/static/img/`
- Naming: `feature-name-screenshot-tr.png`
- Markdown usage: `![Description](/img/feature-name.png)`
- Prefer `.webp` or compressed `.png` when possible

## Important Notes

- **No i18n:** The site is single-locale (Turkish only). There is no `i18n/` directory.
- **No versioning:** The site uses a single unversioned `docs/` folder. There is no `versioned_docs/` or `versioned_sidebars/` directory.
- **Broken links:** `onBrokenLinks` and `onBrokenMarkdownLinks` are set to `ignore` in config. Be careful to maintain valid internal links manually.
- **Edit URL:** "Bu sayfayı düzenle" links point to `https://github.com/MaxHandMade/mhm-rentiva-docs/tree/main/`
- **Search:** Local search plugin (`@easyops-cn/docusaurus-search-local`) is enabled for Turkish content.
