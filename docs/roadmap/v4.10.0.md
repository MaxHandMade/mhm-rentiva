# MHM Rentiva v4.10.0 Roadmap
## Frontend Block Hardening & Contract Stabilization Release

---

## 1. Context & Strategic Positioning

v4.9.9 completed full stabilization (M1–M5), governance hardening, performance cleanup, and meta standardization.  
The system is vertically stable and production-safe.

v4.10.0 is **NOT** a feature expansion release.

It is a **frontend surface hardening release** focused exclusively on:

- Shortcode ↔ Gutenberg Block parity
- Attribute determinism
- Multi-instance safety
- Editor performance stability
- UI contract compliance
- Asset dependency correctness

This release prepares the system for a future UX-focused 4.11.0 roadmap.

---

## 2. Release Principles (Non-Negotiable Constraints)

- No runtime breaking changes.
- No wrapper or slot renaming.
- No REST namespace changes.
- No database schema modifications.
- Migration scripts retained until v4.10.x.
- M4 hard gates enforced (Plugin Check + PHPCS + PHPUnit).
- Backward compatibility with all existing shortcodes is mandatory.

---

## 3. Strategic Intent

v4.10.0 is about three pillars:

> **Parity. Determinism. Stability.**

At the end of this release:

- Every Gutenberg block must behave identically to its shortcode counterpart.
- Attribute handling must be centralized and predictable.
- Multi-instance rendering must be collision-free.
- Editor preview must be stable and performant.
- Asset loading must be deterministic and minimal.

---

# EPIC 1 — Shortcode ↔ Block Parity Enforcement

## Problem

- Inconsistent attribute mapping between block JSON and shortcode parameters.
- Manual camelCase → snake_case conversions scattered across classes.
- Some visibility toggles not fully respected by templates.
- Default value mismatches between block and shortcode usage.

This creates technical debt and hidden regressions.

---

## Scope

### 1.1 Centralized Attribute Mapping Layer

- Implement a canonical attribute mapper.
- Enforce:
  - camelCase → snake_case normalization
  - Boolean normalization
  - Default injection alignment
- Remove ad-hoc per-class mapping logic.

### 1.2 Visibility Contract Enforcement

All vehicle-related blocks must respect canonical template toggles:

- showTitle
- showImage
- showPrice
- showFeatures
- showBookButton
- showRating

No per-block override logic allowed.

### 1.3 Default Parity Matrix

Create a parity matrix documenting:

| Block | Shortcode | Defaults Match | Verified |
|--------|-----------|----------------|----------|

Every block must match its shortcode defaults exactly.

---

## Non-Goals

- No new shortcode.
- No new block.
- No visual redesign.
- No template markup change.
- No CSS renaming.

---

## Acceptance Criteria

- 100% block ↔ shortcode default parity.
- Single canonical attribute mapper in use.
- All visibility toggles verified working.
- No duplicate attribute logic remains.
- PHPUnit full pass.
- PHPCS strict: 0 errors.
- Plugin Check release gate: 0 errors.

---

# EPIC 2 — Block Editor Maturity & Multi-Instance Safety

## Problem

- ServerSideRender previews can be heavy.
- Some blocks may rely on implicit DOM assumptions.
- Multi-instance usage may cause JS selector collisions.
- Inspector controls inconsistent across blocks.

---

## Scope

### 2.1 Multi-Instance Collision Audit

Audit the following blocks:

- vehicles-grid
- vehicles-list
- search-results
- featured-vehicles
- unified-search

Verify:

- No global ID assumptions
- No shared DOM selectors without scoping
- No state leakage across instances

### 2.2 Inspector Panel Standardization

Enforce strict panel structure:

1. Query
2. Filtering
3. Layout & Style
4. Visibility

No duplicated controls.  
No rogue panels.  
No inconsistent naming.

### 2.3 Editor Performance Optimization

- Add debounce to SSR where needed.
- Prevent attribute flood re-rendering.
- Ensure preview rendering is deterministic.

---

## Non-Goals

- No React rewrite.
- No client-side rendering migration.
- No block UI redesign.
- No Gutenberg API experimental features.

---

## Acceptance Criteria

- Multi-instance smoke test passes.
- No console errors in block editor.
- No selector collisions detected.
- SSR preview stable.
- Asset loading unchanged in frontend.
- PHPUnit + Plugin Check pass.

---

# EPIC 3 — Frontend Asset & CSS Contract Stabilization

## Problem

- Potential overlap between legacy rv-* and mhm-rentiva-* CSS hooks.
- Dependency graphs not explicitly audited.
- Risk of silent asset duplication.

---

## Scope

### 3.1 Block Asset Dependency Audit

For each block:

- Verify CSS dependency correctness.
- Verify no duplicate enqueue.
- Verify conditional loading behavior.

### 3.2 CSS Contract Classification

Classify all frontend hooks:

- Stable
- Semi-stable
- Internal

Document findings and ensure no block depends on internal markup depth.

### 3.3 Asset Leak Scan

- Confirm no global enqueue of block-specific CSS.
- Confirm dependency chain remains declarative.

---

## Non-Goals

- No CSS redesign.
- No theme.json refactor.
- No visual refactoring.

---

## Acceptance Criteria

- No duplicate CSS loads.
- No unnecessary frontend asset.
- CSS hook usage documented.
- No internal DOM depth reliance detected.
- Query count delta remains within baseline threshold.

---
