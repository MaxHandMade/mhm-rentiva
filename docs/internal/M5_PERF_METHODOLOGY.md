# M5 — Performance Audit Methodology

**Context:** MHM Rentiva v1.3.5
**Goal:** Establish a rigorous performance baseline after M1 (Shortcode Engine) and M2 (Test Infrastructure) stablization.

## 1. Core Metrics (KPIs)

We will measure the following Key Performance Indicators (KPIs) for critical rendering paths:

### 1.1 Server-Side (PHP/MySQL)
*   **Generate Time (Time to First Byte - Internal):** Time taken by WordPress to process the request and generate HTML.
*   **Memory Usage:** Peak memory allocated during the request.
*   **Database Queries:** Total number of SQL queries executed.
*   **Duplicate Queries:** Number of identical queries (N+1 indicator).
*   **Slow Queries:** Any query taking >50ms.

### 1.2 Client-Side (Asset Weight)
*   **CSS Size:** Total Transfer / Decoded size of stylesheets.
*   **JS Size:** Total Transfer / Decoded size of scripts.
*   **Request Count:** Number of HTTP requests (CSS/JS/Images) generated by the plugin.
*   **Layout Shift (CLS):** Visual stability impact of shortcode injection.

## 2. Tools & Environment

### 2.1 Tooling
*   **Query Monitor (Plugin):** Primary tool for real-time profiling of PHP execution, DB queries, and hooks.
*   **WP-CLI (`wp profile`):** Used for headless benchmarking of specific shortcode execution if needed.
*   **Chrome DevTools (Network/Performance Tabs):** Used for asset weight and LCP/CLS measurement.

### 2.2 Environment
*   **Local Development (XAMPP/Windows):**
    *   *Note:* Absolute timings (ms) will be higher than production.
    *   *Focus:* **Relative comparisons** and **Query Counts** (which are environment-agnostic).

## 3. Scope of Measurement

We will audit the following high-impact scenarios:

1.  **Booking Form (Shortcode):** `[rentiva_booking_form]`
    *   *Critical Path:* Conversion funnel entry point.
2.  **Availability Calendar (Shortcode):** `[rentiva_availability_calendar]`
    *   *Critical Path:* Heavy AJAX and logic load.
3.  **Vehicle Search Results (Page):** `[rentiva_search_results]`
    *   *Critical Path:* High database load (filtering, meta queries).
4.  **Vehicle Detail Page (Block/Shortcode Hybrid):** Single Vehicle View.
    *   *Critical Path:* Asset loading correctness (M1 parity check).

## 4. Execution Plan (Step 2)

For each scenario:
1.  Load the page ~3 times to warm object cache.
2.  Record "Median" values for Generate Time and Memory.
3.  Record "Constant" values for Query Counts.
4.  Capture Screen of Query Monitor summary (or log text).
5.  Document in `docs/internal/M5_BASELINE_REPORT.md`.
